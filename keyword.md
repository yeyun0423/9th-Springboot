# 스프링 핵심 개념 정리

## SOLID 원칙
1. **S – 단일 책임 원칙 (Single Responsibility Principle, SRP)**
   - 클래스는 **하나의 책임만** 가져야 한다.
   - 변경의 이유가 **오직 하나**여야 한다.

2. **O – 개방-폐쇄 원칙 (Open/Closed Principle, OCP)**
   - 소프트웨어 요소는 **확장에는 열려 있고, 수정에는 닫혀 있어야 한다.**

3. **L – 리스코프 치환 원칙 (Liskov Substitution Principle, LSP)**
   - **자식 클래스는 부모 클래스를 대체**할 수 있어야 한다.

4. **I – 인터페이스 분리 원칙 (Interface Segregation Principle, ISP)**
   - 사용하지 않는 기능에 의존하지 않도록 **인터페이스를 작게 분리**해야 한다.

5. **D – 의존 역전 원칙 (Dependency Inversion Principle, DIP)**
   - 고수준 모듈은 저수준 모듈에 의존하지 말고, **추상화에 의존**해야 한다.


## DI (Dependency Injection, 의존성 주입)
- 객체가 다른 객체를 사용할 때, **직접 생성하지 않고 외부에서 주입받는 방식**.
- 스프링에서는 **생성자, 세터, 필드** 등을 통해 주입 가능.


## IoC (Inversion of Control, 제어의 역전)
- 객체의 생성과 의존성 관리를 **개발자가 아닌 스프링 컨테이너가 대신 관리**.
- 프로그램의 제어 흐름을 직접 제어하지 않고, **컨테이너가 관리한다는 의미**.

## 생성자 주입 vs 수정자 주입 vs 필드 주입

### 1. 생성자 주입 (Constructor Injection)
- **정의:** 생성자를 통해 의존성을 주입.
- **특징**
  - `final` 키워드 사용 가능 → **불변성 보장**.
  - 주입을 누락하면 객체 생성 불가 → **안전성** 높음.
  - 스프링에서 **권장하는 방식**.


### 2. 수정자 주입 (Setter Injection)
- **정의:** 세터 메서드를 통해 의존성을 주입.
- **특징**
  - 객체 생성 후 의존성 변경 가능.
  - **선택적 의존성**(필수가 아닌 경우)에 적합.
  - 런타임 중에 변경될 수 있어 **불변성 보장 어려움**.


### 3. 필드 주입 (Field Injection)
- **정의:** 필드에 직접 `@Autowired` 등을 붙여 주입.
- **특징**
  - 코드가 간단하고 편리.
  - 그러나 **테스트 및 유지보수에 불리**.
  - 스프링 컨테이너 외부에서 사용 어려움 → **권장되지 않음**.


## AOP (Aspect-Oriented Programming, 관점 지향 프로그래밍)
- **핵심 기능과 부가 기능을 분리하여 모듈화**하는 프로그래밍 기법.
- 로깅, 트랜잭션, 보안 같은 **부가 기능을 핵심 로직과 분리** → 코드 중복 감소, 유지보수 용이.
- 스프링에서는 **프록시 기반**으로 AOP 지원.


## 서블릿 (Servlet)
- 자바를 이용해 **웹 요청과 응답을 처리하는 서버 측 프로그램**.
- 클라이언트의 요청(HTTP)을 받아 처리 후, 결과를 응답으로 반환.
- **자바 EE 표준 스펙**이며, 스프링 MVC도 내부적으로 서블릿 기반으로 동작.
